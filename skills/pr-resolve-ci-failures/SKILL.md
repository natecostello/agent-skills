Resolve CI failures on pull requests.

Argument: $ARGUMENTS (optional PR number, e.g. "42". If empty, detect from current branch or process all open PRs)

## Instructions

You are resolving CI failures on pull requests. Follow these steps carefully:

**Attribution rule:** Every comment or message you post on GitHub is posted under the repo owner's account. You MUST append the following attribution line to ALL GitHub-visible text so readers know it was generated by an AI agent, not written by the human account holder:

```
🤖 Generated with [Claude Code](https://claude.ai/code)
```

### Step 1: Identify the PR(s)

If a PR number was provided in the argument ("$ARGUMENTS"), use that:
```
gh pr view $ARGUMENTS --json number,headRefName,statusCheckRollup
```

If no argument was provided (empty string), try the current branch first:
```
gh pr view --json number,headRefName,statusCheckRollup
```

If that fails (no PR for current branch), find all open PRs:
```
gh pr list --state open --json number,headRefName,title
```

Present the list and ask which PR(s) to process, or process all if the user confirms.

### Step 2: Fetch CI check status

For each PR, get the check runs and their statuses:
```
gh pr checks {pr_number}
```

**IMPORTANT:** Do NOT use `--json` with `gh pr checks` — it silently returns empty results in some versions of `gh`. Use the plain text output and parse the tab-separated columns (name, status, duration, url).

Filter to checks with status `fail`. Skip checks that show `pass`, `skipped`, or `pending`.

If all checks pass, report "All CI checks passing" and move to the next PR.

### Step 3: Fetch failure logs

For each failing check:

1. **Get the run ID** from the check details:
   ```
   gh api repos/{owner}/{repo}/actions/runs --jq '.workflow_runs[] | select(.head_sha == "{head_sha}") | {id, name, conclusion}'
   ```

2. **Get the failed job(s)** within the run:
   ```
   gh api repos/{owner}/{repo}/actions/runs/{run_id}/jobs --jq '.jobs[] | select(.conclusion == "failure") | {id, name, conclusion}'
   ```

3. **Download the job logs**:
   ```
   gh api repos/{owner}/{repo}/actions/jobs/{job_id}/logs
   ```

4. **Extract the relevant failure section** — look for error messages, failed test output, lint violations, or type errors. The full log may be very long; focus on the last ~200 lines of the failing step, or search for patterns like:
   - `FAILED` / `ERROR` / `error:` / `Error:`
   - `FAIL:` (pytest)
   - `error[` (ruff/mypy)
   - Exit codes != 0

### Step 4: Diagnose each failure

For each failing check, categorize the failure:

1. **Lint failure** (ruff, eslint, etc.) — read the exact violations (file, line, rule)
2. **Type check failure** (mypy, pyright, tsc) — read the exact errors (file, line, type mismatch)
3. **Test failure** (pytest, jest, etc.) — read the failing test name, assertion error, traceback
4. **Format failure** (ruff format, prettier) — identify which files need reformatting
5. **Build failure** — read the compiler/bundler error
6. **Dependency failure** — missing package, version conflict, lock file mismatch
7. **Infrastructure/flaky** — timeout, network error, runner issue (not fixable from code)

For category 7 (infrastructure/flaky), skip with a note to the user — these require re-running the workflow, not code changes.

### Step 5: Fix the failures

1. Check out the PR's branch (use an existing worktree if available):
   ```
   git checkout {headRefName}
   git pull
   ```

2. For each fixable failure:
   - **Read the failing file(s)** to understand context
   - **Apply the fix** — be precise and minimal. Only change what's needed to fix the CI failure.
   - **Run the check locally** if possible to verify the fix before pushing:
     - Lint: `uv run ruff check {files}` or equivalent
     - Format: `uv run ruff format {files}` or equivalent
     - Type check: `uv run mypy {files}` or equivalent
     - Tests: `uv run pytest {test_file}::{test_name}` or equivalent
   - **Do NOT** introduce unrelated changes, refactors, or "improvements" while fixing CI

3. If a failure cannot be fixed without significant code changes or architectural decisions, skip it and report to the user.

### Step 6: Commit and push

**Commit per CI check fixed** — create a separate commit for each distinct CI failure addressed:

```
ci: fix <check-name> — <short description>

Resolves <check-name> failure on PR #{pr_number}:
- <what was changed and why>

Verified locally:
- <command run and result, e.g. "ruff check src/ — 0 violations">

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

Rules:
- `<check-name>` is the CI check name (e.g. `lint`, `typecheck`, `test`)
- Group related violations into a single commit (e.g. all ruff fixes in one commit, not one per file)
- Stage only the files relevant to that specific failure before committing
- Always verify locally before committing if the tool is available

Push to the branch:
```
git push
```

### Step 7: Verify CI re-run

After pushing, check if CI has started re-running:
```
gh pr checks {pr_number}
```

Do NOT wait for CI to complete — just confirm the push triggered new runs.

### Step 8: Report

Print a summary:
- Total failing checks found
- Checks fixed (with check name and brief description of fix)
- Checks skipped (with reason: infrastructure/flaky, requires human decision, too risky)
- Local verification results for each fix
- Any errors encountered

If checks were skipped, suggest the user review them manually or re-run the workflow for flaky failures.
