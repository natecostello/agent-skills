Resolve review comments on pull requests.

Argument: $ARGUMENTS (optional PR number, e.g. "42". If empty, detect from current branch or process all open PRs)

## Instructions

You are resolving PR review comments. Follow these steps carefully:

### CRITICAL RULES — Do NOT skip these

1. **Always display a summary before acting.** After fetching threads, display a numbered table of ALL unresolved threads (file, line, author, first 80 chars, your proposed action: fix/skip/reply). Then proceed — do NOT wait for user approval unless something looks risky or ambiguous.
2. **Every thread resolution requires a thread reply.** Before resolving ANY thread (whether by code fix or by declining the suggestion), you MUST post an `in_reply_to` reply on that thread explaining what you did and why. This creates a visible audit trail under the original comment.
3. **NEVER post standalone PR comments.** All responses to review feedback must be posted as `in_reply_to` replies on the specific thread, not as top-level PR comments. Standalone comments have no link to the original feedback.
4. **NEVER resolve a thread silently.** Resolving without a reply makes it impossible for others to understand why the thread was closed.
5. **Skipping requires justification.** If you skip a thread (decline to implement the suggestion), your thread reply MUST include a clear rationale. The summary table shown before acting gives the user a chance to intervene before you proceed.

**Attribution rule:** Every comment, reply, or message you post on GitHub (PR comments, thread replies, sub-PR close comments) is posted under the repo owner's account. You MUST append the following attribution line to ALL GitHub-visible text so readers know it was generated by an AI agent, not written by the human account holder:

```
🤖 Generated with [Claude Code](https://claude.ai/code)
```

### Step 0: Wait for Copilot Review (if pending)

Before fetching comments, check whether a Copilot review has already been submitted. Use the review request state to decide how long to wait.

**IMPORTANT — REST, not GraphQL:** Use the REST API (`gh api repos/.../reviews`) for review detection, NOT GraphQL with `$`-prefixed variables. GraphQL variable names (`$owner`, `$repo`, `$pr`) can pick up invisible Unicode artifacts when the query passes through conversation context, causing `Expected VAR_SIGN, actual: UNKNOWN_CHAR` errors that silently return error JSON instead of review data.

1. **Check for existing Copilot reviews** using the REST API:
   ```bash
   gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews --jq '[.[] | select(.user.login == "copilot-pull-request-reviewer[bot]")] | length'
   ```
   Store this count as `previous_review_count`.

   **If count > 0, validate the latest review is not an error:**
   ```bash
   gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews --jq '[.[] | select(.user.login == "copilot-pull-request-reviewer[bot]")] | last | .body'
   ```
   If the body contains "encountered an error" or "unable to review", the review is an error — treat it as if no review exists (do NOT skip to Step 1). Re-request a review in step 2 and poll for a new one whose count exceeds `previous_review_count`.

   If the latest review body does NOT contain error language, a valid review exists — skip to Step 1.

2. **Check if Copilot is a requested reviewer** (i.e., queued but hasn't responded yet):
   ```bash
   gh api repos/{owner}/{repo}/pulls/{pr_number}/requested_reviewers --jq '[.users[] | select(.login == "Copilot")] | length'
   ```
   - If Copilot is NOT in requested reviewers and has no reviews, **request it**:
     ```bash
     gh pr edit {pr_number} --add-reviewer "@copilot"
     ```

3. **Determine how long Copilot has already been waiting** using the timeline API:
   ```bash
   gh api repos/{owner}/{repo}/issues/{pr_number}/timeline --paginate \
     --jq '[.[] | select(.event == "review_requested" and .requested_reviewer.login == "Copilot")] | last | .created_at'
   ```
   This returns the timestamp of the most recent Copilot review request. Calculate `elapsed_minutes` = now - that timestamp. Subtract `elapsed_minutes` from the 20-minute budget to get `remaining_wait`. If Copilot was requested 15 minutes ago, only poll for ~5 more minutes instead of a full 20.

   If no review_requested event is found (e.g., you just requested it in step 2), use `remaining_wait` = 20 minutes.

4. **Poll for the review** every 30 seconds (up to `remaining_wait` minutes, max 20) using a background agent:
   ```bash
   gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews --jq '[.[] | select(.user.login == "copilot-pull-request-reviewer[bot]")] | length'
   ```
   The review has landed when the count exceeds `previous_review_count`.

5. **Also check for Copilot sub-PRs** (Copilot sometimes delivers feedback as sub-PRs):
   ```bash
   gh pr list --state open --json number,headRefName,title --search "head:copilot/sub-pr-{pr_number}"
   ```

6. **If the wait budget expires with no review**, check the request state to decide next action:
   - **Copilot still in `requested_reviewers`:** The review is queued but GitHub is slow. Flag to user: "Copilot review was requested {elapsed_minutes} minutes ago but hasn't responded. Options: (a) wait longer, (b) proceed without Copilot review."
   - **Copilot NOT in `requested_reviewers` and no review:** Copilot may have declined or isn't available for this repo. Proceed and note in the final report.

**NOTE on Copilot review timing:** Large diffs (500+ lines) routinely take 12-24 minutes for Copilot to review. The 20-minute window accounts for this. Do NOT use shorter timeouts.

### Step 1: Identify the PR(s)

If a PR number was provided in the argument ("$ARGUMENTS"), use that:
```
gh pr view $ARGUMENTS --json number,headRefName
```

If no argument was provided (empty string), try the current branch first:
```
gh pr view --json number,headRefName
```

If that fails (no PR for current branch), find all open PRs:
```
gh pr list --state open --json number,headRefName,title
```

Present the list and ask which PR(s) to process, or process all if the user confirms.

### Step 2: Fetch all review comments

For each PR, fetch the review comments:
```
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments --paginate
```

Also fetch review threads to determine which are resolved:
```
gh api graphql -f query='query { repository(owner: "{owner}", name: "{repo}") { pullRequest(number: {pr_number}) { reviewThreads(first: 100) { nodes { id isResolved comments(first: 10) { nodes { id body path line originalLine diffHunk author { login } } } } } } } }'
```

**IMPORTANT:** Use inline values in the GraphQL query (as shown above), NOT `$`-prefixed variables with `-f`/`-F` flags. GraphQL variable names can pick up invisible Unicode artifacts from conversation context, causing silent query failures.

Extract the owner and repo from the remote:
```
gh repo view --json owner,name -q '.owner.login + "/" + .name'
```

### Step 2b: Check for Copilot sub-PRs (review-as-PR pattern)

GitHub Copilot sometimes delivers review feedback as sub-PRs instead of inline comments. Check for these:

```
gh pr list --state open --json number,headRefName,title,body --search "head:copilot/sub-pr-{pr_number}"
```

If any open PRs exist with branch name `copilot/sub-pr-{pr_number}`:

1. **Fetch the diff** for each sub-PR: `gh pr diff {sub_pr_number}`
2. **Read the sub-PR body** to understand the rationale
3. **Treat each sub-PR as an actionable review suggestion** — evaluate it the same way you would a suggestion block in an inline comment:
   - Does the change make sense? Is the rationale sound?
   - Does it conflict with other review comments or the project's conventions?
4. **If you agree:** apply the change directly to the parent PR's branch (not via merge), commit with the standard format referencing the sub-PR, and close the sub-PR with a comment explaining it was incorporated
5. **If you disagree or it's ambiguous:** skip it and report to the user, leaving the sub-PR open

Commit format for sub-PR changes:
```
fix: <short description>

Addresses Copilot review of PR #{pr_number} (delivered as sub-PR #{sub_pr_number}):
- <what was changed>

Remaining unresolved:
- <anything not addressed, or 'None'>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

When closing the sub-PR:
```
gh pr close {sub_pr_number} --comment "Closing — this fix has been applied directly to PR #{pr_number}'s branch in commit {sha}. <brief rationale for accepting>.

🤖 Generated with [Claude Code](https://claude.ai/code)"
```

### Step 3: Filter to unresolved threads

From the GraphQL response, keep only threads where `isResolved` is `false`. Skip threads that are already resolved.

For each unresolved thread, extract:
- The file path and line number from the first comment
- The comment body (the review feedback)
- The thread ID (needed to resolve it later)
- The diff hunk for context

### Step 4: Display thread summary

Before making changes, display a table of all unresolved threads so the user can see what you're about to do:

```
| # | File:Line | Author | Summary (first 80 chars) | Proposed action |
|---|-----------|--------|--------------------------|-----------------|
| 1 | src/foo.py:42 | copilot | "Add error handling for..." | Fix — apply suggestion |
| 2 | src/bar.py:10 | copilot | "Consider extracting..." | Skip — descriptive, no concrete change |
| 3 | src/baz.py:99 | copilot | "Refactor this into a..." | Defer — valid but too large for this PR |
```

For each thread marked "Skip" or "Defer", briefly explain why. Then proceed with fixes. The user can interrupt if they disagree.

### Step 5: Process each unresolved comment

For each unresolved thread (after user approves the plan):

1. **Read the current file** at the path mentioned in the comment
2. **Understand the suggestion** — parse the comment body. Look for:
   - GitHub suggestion blocks (```suggestion ... ```) — these are exact code replacements
   - Descriptive feedback ("rename this", "add error handling", "this should be X not Y")
   - Questions or discussion (not actionable — skip these with a note)
3. **Check if the suggestion is valid** — does it make sense given the current code? Is the line still present (it may have been changed since the review)?
4. **Implement the fix** if the suggestion is actionable:
   - For GitHub suggestion blocks: apply the exact replacement
   - For descriptive feedback: read surrounding code, understand the intent, make the change
   - For ambiguous or large-scope suggestions: consider deferring (see below)
5. **Skip** if:
   - The comment is a question or discussion, not an actionable request
   - The suggested change conflicts with other comments
   - The file or line no longer exists
6. **Defer** if:
   - The suggestion is valid but too large or risky for this PR
   - It would require architectural changes beyond the PR's scope
   - It's a good improvement but not related to the PR's purpose

   For each deferred thread, **create a GitHub issue** to track the work:
   ```bash
   gh issue create --title "<short description from the review comment>" --body "$(cat <<'EOF'
   ## Context

   From Copilot review on PR #{pr_number}, `{file}` line {line}:

   > {review comment body, trimmed to key points}

   ## Suggested change

   {description of what the reviewer suggested}

   ## Why deferred

   {one-line rationale for deferring}

   ---
   *Created from PR #{pr_number} review comment.*
   EOF
   )"
   ```

   Then reply to the thread with the issue link (see Step 7 reply format for "Deferred").

### Step 6: Commit and push

After processing all comments for a PR:

1. Check out the PR's branch:
   ```
   git checkout {headRefName}
   git pull
   ```
2. **Commit per review comment** — create a separate commit for each review comment addressed. Each commit must reference the specific comment using this format:
   ```
   fix: <short description>

   Addresses review comment by <reviewer> on <file> lines <N-M>:
   - <what was changed>

   Remaining unresolved:
   - <anything not yet addressed, or 'None'>

   Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
   ```

   Rules:
   - `<reviewer>` is the GitHub username of the comment author
   - `<file>` is the path from the review comment (e.g. `src/postcard/engines/diffbrush.py`)
   - `<N-M>` is the line range from the comment; use a single number if it's one line
   - List each concrete change made under the "Addresses" section
   - If the commit does NOT fully resolve every point in the comment, the "Remaining unresolved" section MUST list what was not addressed and why
   - If everything is resolved, write `- None`
   - Stage only the files relevant to that specific comment before committing

   Example:
   ```
   fix: use explicit type annotation for response dict

   Addresses review comment by copilot on src/postcard/pipeline/fulfillment.py lines 19-20:
   - Changed bare `dict` return type to `dict[str, Any]`
   - Added `from typing import Any` import

   Remaining unresolved:
   - None

   Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
   ```

3. Push to the branch:
   ```
   git push
   ```

### Step 7: Reply to threads, THEN resolve

**Every thread must get a reply before it is resolved.** This is the audit trail that links your action to the original feedback.

For each thread, FIRST post an `in_reply_to` reply explaining what was done.

**Use the REST API for replies** (NOT the GraphQL `addPullRequestReviewComment` mutation — it requires a `pullRequestReviewThreadId` parameter that doesn't exist in the schema and will silently fail):

```
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments \
  -f "body=<your reply>

🤖 Generated with [Claude Code](https://claude.ai/code)" \
  -F "in_reply_to={comment_id}"
```

The `comment_id` is the numeric REST API ID of the original review comment (from `gh api repos/{owner}/{repo}/pulls/{pr_number}/comments --jq '.[].id'`), NOT the GraphQL node ID. Do NOT pass a `commit_id` parameter — it's not needed for reply comments and will error if the commit SHA doesn't match exactly.

Reply content depends on the action taken:
- **Fixed:** "Applied this suggestion in commit {sha}: {brief description of change}."
- **Skipped (not actionable):** "This is descriptive feedback rather than a concrete change. Noted for future work — not implementing in this PR."
- **Skipped (too large):** "This would require {description of scope}. Deferring to a follow-up PR."
- **Declined:** "Not implementing because {specific rationale}."
- **Deferred:** "Valid suggestion — created #{issue_number} to track this. Deferring from this PR because {rationale}."

THEN resolve the thread (for both fixes and intentional skips that the user approved):
```
gh api graphql -f query='
mutation($threadId: ID!) {
  resolveReviewThread(input: {threadId: $threadId}) {
    thread { isResolved }
  }
}' -f threadId='{thread_id}'
```

Do NOT resolve threads where the user has not yet approved the skip.

### Step 7b: Verify replies were posted

After posting all replies and resolving threads, **verify** that every resolved thread actually has a reply from the current user. This catches silent failures (wrong mutation API, auth errors, etc.):

```
gh api graphql -f query='query { repository(owner: "{owner}", name: "{repo}") { pullRequest(number: {pr_number}) { reviewThreads(first: 100) { nodes { id isResolved comments(last: 2) { nodes { author { login } body } } } } } } }' --jq '.data.repository.pullRequest.reviewThreads.nodes[] | select(.isResolved == true) | {id: .id, last_author: .comments.nodes[-1].author.login, has_reply: (.comments.nodes | length > 1)}'
```

For each resolved thread, check:
- `has_reply` is true (more than just the original review comment)
- `last_author` is NOT the reviewer (e.g. not `copilot-pull-request-reviewer`) — it should be the repo owner's account

If any resolved thread is missing a reply:
1. **Unresolve it** (if possible) or flag it clearly in the report
2. **Re-post the reply** using the REST API: `gh api repos/{owner}/{repo}/pulls/{pr_number}/comments -f body='...' -F in_reply_to={comment_id}`
3. **Re-resolve** after confirming the reply posted

This is critical because GitHub's GraphQL `resolveReviewThread` mutation succeeds even when the reply mutation silently fails, leaving resolved threads with no visible explanation.

### Step 8: Re-check loop (post-push monitoring)

**If any commits were pushed in Step 6**, a new Copilot review may be triggered. This step ensures all new comments are resolved before reporting completion.

1. **Immediately request a new Copilot review** after pushing:
   ```
   gh pr edit {pr_number} --add-reviewer "@copilot"
   ```
   Store the current Copilot review count as `previous_review_count`.

2. **Poll for the new review** every 30 seconds (up to **20 minutes**) using a background agent:
   ```
   gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews --jq '[.[] | select(.user.login == "copilot-pull-request-reviewer[bot]")] | length'
   ```
   A new review has landed when the count exceeds `previous_review_count`.

3. **If 20 minutes elapse with no new review**, check request state:
   - **Copilot still in `requested_reviewers`:** Flag to user: "Copilot review requested after push but hasn't responded after 20 minutes. Options: (a) wait longer, (b) proceed without additional review."
   - **Copilot NOT in `requested_reviewers`:** Proceed — Copilot may have declined to re-review.

4. Once a new review lands, re-fetch threads using the Step 2 GraphQL query (with inline values) to check for new unresolved threads.

5. **If new unresolved threads exist**, go back to Step 3 (filter to unresolved) and repeat the full cycle: display summary → process comments → commit → push → reply → resolve → verify → re-check.

6. **If no new unresolved threads appear** after the wait period, proceed to Step 9 (Report).

7. **Loop limit: 3 rounds maximum.** After 3 rounds of push → re-check → resolve, stop and report any remaining unresolved threads to the user. This prevents infinite loops if Copilot keeps finding new issues.

Track the round number and display it in the summary table header:
```
### Round 2: New comments after push
| # | File:Line | Author | Summary | Proposed action |
```

**NOTE on review timing:** Copilot re-reviews after a push typically take 10-15 minutes. The 20-minute window accounts for large diffs and GitHub-side queuing delays.

### Step 9: Report

Print a summary:
- Total rounds completed (e.g. "2 rounds: initial + 1 re-check")
- Total unresolved threads found (across all rounds)
- Threads resolved (with file:line and brief description of fix)
- **Threads verified** (confirm reply count matches resolved count)
- Threads skipped (with reason: question/discussion, ambiguous, conflict, file changed, too risky)
- Threads deferred (with issue number and rationale for each)
- Copilot sub-PRs found, accepted (with commit sha), or skipped (with reason)
- Any errors encountered

If comments or sub-PRs were skipped, suggest the user review them manually.
