Resolve merge conflicts on pull requests.

Argument: $ARGUMENTS (optional PR number, e.g. "42". If empty, detect from current branch or process all open PRs)

## Instructions

You are resolving merge conflicts on pull requests. Follow these steps carefully:

### Step 1: Identify the PR(s)

If a PR number was provided in the argument ("$ARGUMENTS"), use that:
```
gh pr view $ARGUMENTS --json number,headRefName,baseRefName,mergeable,title
```

If no argument was provided (empty string), try the current branch first:
```
gh pr view --json number,headRefName,baseRefName,mergeable,title
```

If that fails (no PR for current branch), find all open PRs and check which have conflicts:
```
gh pr list --state open --json number,headRefName,baseRefName,mergeable,title
```

Filter to PRs where `mergeable` is `"CONFLICTING"`. If no PRs have conflicts, report "No merge conflicts found" and stop.

### Step 2: Understand the PR context

For each conflicting PR:

1. **Read the PR description and commits** to understand the intent of the branch:
   ```
   gh pr view {pr_number} --json body,commits
   ```

2. **Check what the base branch is** (usually `main`) and ensure it's up to date:
   ```
   git fetch origin {baseRefName}
   ```

3. **Read the commit log** for the PR branch to understand the history of changes:
   ```
   gh pr view {pr_number} --json commits --jq '.commits[].messageHeadline'
   ```

### Step 3: Check out the branch and attempt merge

1. Check out the PR's branch (use an existing worktree if available):
   ```
   git checkout {headRefName}
   git pull
   ```

2. Attempt to merge the base branch to surface the conflicts:
   ```
   git merge origin/{baseRefName}
   ```

3. List the conflicted files:
   ```
   git diff --name-only --diff-filter=U
   ```

### Step 4: Analyze each conflict

For each conflicted file:

1. **Read the file** to see the conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`)
2. **Understand both sides:**
   - `<<<<<<< HEAD` (ours) — the PR branch's version. What was the intent of this change?
   - `>>>>>>> origin/{baseRefName}` (theirs) — the base branch's version. What changed on the base?
3. **Read the git log for both sides** to understand why each change was made:
   ```
   git log --oneline HEAD -- {file}
   git log --oneline origin/{baseRefName} -- {file}
   ```

### Step 5: Resolve each conflict

For each conflicted file, determine the correct resolution:

1. **Non-overlapping changes** — both sides changed different parts of the same region. Keep both changes, merged correctly.
2. **Same intent, different implementation** — both sides made the same fix. Keep whichever is more complete; discard the duplicate.
3. **Contradictory changes** — the two sides disagree on what the code should be. Prefer the PR branch's intent (since that's the active work), but incorporate any necessary updates from base (new imports, renamed symbols, updated APIs).
4. **Structural conflicts** (e.g., file renamed on one side, modified on the other) — follow the rename and apply the modifications to the new location.
5. **Lock file conflicts** (`uv.lock`, `package-lock.json`, etc.) — do NOT manually resolve. Instead, after resolving all code conflicts:
   ```
   # For uv.lock:
   git checkout --theirs uv.lock
   uv lock
   git add uv.lock

   # For package-lock.json:
   git checkout --theirs package-lock.json
   npm install
   git add package-lock.json
   ```

**When in doubt:** If a conflict involves complex logic where the correct resolution isn't clear, skip it and report to the user. Do NOT guess on semantic conflicts that could introduce bugs.

After resolving each file:
```
git add {file}
```

### Step 6: Verify the resolution

After all conflicts are resolved:

1. **Check no conflict markers remain:**
   ```
   grep -r '<<<<<<< ' --include='*.py' --include='*.ts' --include='*.js' --include='*.md' .
   ```

2. **Run available checks locally** to verify the merge didn't break anything:
   - Lint: `uv run ruff check src/ tests/` or equivalent
   - Type check: `uv run mypy src/` or equivalent
   - Tests: `uv run pytest` or equivalent
   - Format: `uv run ruff format --check src/ tests/` or equivalent

3. If any check fails, fix the issue before committing. The merge resolution may have introduced errors (e.g., duplicate imports, missing imports for code that moved).

### Step 7: Commit and push

Complete the merge commit:
```
git commit -m "$(cat <<'EOF'
merge: resolve conflicts with {baseRefName}

Conflicted files:
- <file1> — <brief description of resolution>
- <file2> — <brief description of resolution>

Verified locally:
- <checks run and results>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
EOF
)"
```

Rules:
- Use a single merge commit (not individual commits per file)
- List every conflicted file and how it was resolved
- Include local verification results
- If any conflicts were skipped (left for human review), do NOT commit — abort the merge with `git merge --abort` and report

Push to the branch:
```
git push
```

### Step 8: Report

Print a summary:
- Total PRs with conflicts found
- For each PR:
  - Number of conflicted files
  - Files resolved (with brief description of resolution strategy)
  - Files skipped (with reason: complex semantic conflict, unclear intent, too risky)
  - Local verification results
- Any errors encountered

If conflicts were skipped, suggest the user resolve them manually and explain what makes them ambiguous.
